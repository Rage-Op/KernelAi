# Phase 3 — UI Design Contract

> Visual and interaction contract for the KERNEL Face: the living particle cloud, the Stage choreography, and the widgets. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform note:** This is a **native Swift/SwiftUI + Metal macOS app** (macOS 15 target, macOS 14 min), NOT a web project. "shadcn precision" here is an **aesthetic target emulated in native SwiftUI**, not the shadcn CLI/registry. The shadcn initialization gate and registry-safety vetting are therefore **not applicable** (`Tool: none`). All values below are SwiftUI/Metal-implementable design constants, not Tailwind tokens.
>
> **Scope boundary:** This document is the DESIGN contract — tokens, materials, motion law, choreography behavior, and per-widget visual specs. The TECHNICAL implementation research (MTKView/compute-shader wiring, AVSpeechSynthesizer delegate plumbing, UDS frame shapes, RAM budget) is owned by a separate agent and lives in `CLAUDE.md`. ## Spacing Scale

4-pt grid. Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, chip inner padding, hairline insets |
| sm | 8px | Compact element spacing, intra-row gaps inside a widget |
| md | 16px | Default element spacing, widget inner content gap |
| lg | 24px | Widget content padding (glass card interior) |
| xl | 32px | Gap between two co-focused widgets; widget-to-cloud breathing room |
| 2xl | 48px | Stage vertical inset from canvas edges |
| 3xl | 64px | Full-screen canvas safe-area margin around the cloud |

**Exceptions:**
- Corner pill (Claude Code session state): fixed inset **16px** from top-left screen corner; pill internal padding **12px** horizontal / **8px** vertical (12 is off-grid by design for the compact pill — noted exception).
- Suggested-action chips (mail widget): min height **28px**, horizontal padding **12px** (compact control exception).
- Hit targets for any tappable control: minimum **44×44px** effective touch/click target even when the visual is smaller (Apple HIG accessibility floor).

## Typography

Exactly 4 sizes, exactly 2 weights. SF Pro Display for Display/Heading, SF Pro Text for Label/Body. Tabular numerals (`monospacedDigit()` / `.tabularNumbers`) mandatory on ALL money, balances, counts, and chart axis values.

| Role | Size | Weight | Line Height | Family |
|------|------|--------|-------------|--------|
| Display | 28px | Semibold (600) | 1.2 | SF Pro Display |
| Heading | 20px | Semibold (600) | 1.2 | SF Pro Display |
| Body | 16px | Regular (400) | 1.5 | SF Pro Text |
| Label | 14px | Regular (400) | 1.4 | SF Pro Text |

**Weight rule:** only Regular (400) and Semibold (600) exist in this phase. No light, medium, bold, or black. Emphasis is achieved with weight (semibold) or the accent color, never a third weight.

**Numerals:** every balance, currency amount, transaction value, event count ("3 events"), unread count, and chart tick uses tabular/monospaced digits so figures align vertically and never reflow as they count up.

**Color counter-up:** numbers animate by counting up (see Motion Law) — they must use tabular numerals so width is stable during the count.

## Color

shadcn-dark restraint. Deep spatial black base, low-chroma zinc neutral surfaces, hairline borders, ONE accent only. 60/30/10 split.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#08080A` (near-black, spatial canvas) | Full-screen background canvas behind the cloud; the "infinite space" base |
| Secondary (30%) | zinc neutrals: surface `#18181B` (zinc-900), elevated `#27272A` (zinc-800), muted text `#A1A1AA` (zinc-400) | Frosted-glass widget bodies (material tint over this), pill body, secondary text |
| Accent (10%) | indigo `#7C8CFF` → cyan `#42E8E0` (living gradient, never fixed) | The cloud particles; active/focus states only |
| Destructive | `#F87171` (low-chroma red) | Reserved — no destructive actions ship in Phase 3; declared for Phase 4 email-send/finance forward-compat |

**Hairline borders:** all glass cards, pills, and chips carry a 1px hairline border at **white 6–8% opacity** (`Color.white.opacity(0.07)`). Never a solid or accent border at rest.

**Text on dark (AA on `#08080A` / over glass):**
- Primary text: `#FAFAFA` (zinc-50) — AA+ on all surfaces.
- Secondary/muted text: `#A1A1AA` (zinc-400) — meets AA for ≥16px body on the near-black base; do NOT use zinc-400 for <14px text or over the brightest glass without re-checking contrast.
- Never place body text directly over the brightest part of the live cloud — text always lives on a frosted-glass widget (the glass provides the contrast floor).

**Accent reserved for — EXPLICIT list (accent appears ONLY here):**
1. The particle cloud field (indigo↔cyan living color — the one place the gradient lives full-strength).
2. Active/focus ring on a control or selected chip (a single accent hairline or glow, not a fill).
3. The single brightening flash on a particle burst tied to a TTS boundary crossing.
4. The active timeframe segment label in the spending chart's W/M/Y switcher (accent text or underline, not a filled button).
5. The live-pulse dot on the corner pill while a Claude Code session streams.

Accent is NEVER used for: widget backgrounds, body text, icons at rest, chip fills, primary buttons, dividers, or "all interactive elements." Everything not in the list above is neutral.

## Materials & Depth (SwiftUI-specific extension)

The standard web template has no materials section; this product's identity is glass + depth, so it is contractually specified here.

| Surface | Material | Notes |
|---------|----------|-------|
| Widget body (in focus) | `.ultraThinMaterial` | Frosted glass; near-black canvas + cloud read through it faintly |
| Widget body (denser content: spending chart, email preview) | `.regularMaterial` | Slightly more opaque for chart/text legibility |
| Corner pill | `.regularMaterial` | Compact, must stay legible over a busy cloud |
| Canvas haze | additive volumetric haze layer | Faint, behind/around the cloud — never a flat fill |

**Depth law:**
- Whatever is NOT in focus is **depth-blurred** (Gaussian, ~8–16pt radius) and dimmed (~40–60% opacity). When a widget blooms forward, its forward-blur clears to 0; when it disperses, blur returns as it recedes.
- Subtle parallax: the cloud and haze shift slightly against widget motion to sell infinite-space depth. Parallax is gentle (a few points of offset), never disorienting.
- Layer order (back→front): canvas `#08080A` → volumetric haze → particle cloud → depth-blurred background widgets → in-focus glass widget(s) → corner pill (session state only).

## The Cloud — Centerpiece & Conductor (domain extension)

The cloud is the product's visual signature and the conductor of all choreography. CLOUD-01..03, CLOUD-05.

**Form:** thousands of soft, additively-blended particles forming a breathing nebula centered on the canvas. Spatial-black canvas, volumetric haze, subtle parallax, sense of infinite depth — never a flat screen, never a fixed-shape "orb."

**Color:** lives BETWEEN indigo `#7C8CFF` and cyan `#42E8E0` — a living field that drifts across the range, NOT a fixed gradient. Individual particles sample different points in the indigo↔cyan range; the field as a whole slowly migrates.

**Behavioral states (driven by mic RMS, computed entirely in the Face — never round-tripped through the daemon):**

| State | Particle behavior | Color/brightness |
|-------|-------------------|------------------|
| Idle (no speech) | Gentle drift, slow breathing expansion/contraction | Calm, dimmer, cooler (toward cyan) |
| Speaking (mic amplitude up) | Amplitude pushes particles OUTWARD; field expands with the voice | Brightens; warms toward indigo on peaks |
| Quiet pull (between phrases) | Particles settle/recohere inward | Eases back to calm |
| TTS boundary burst | A localized outward burst + single brighten flash on each boundary crossing (see Choreography) | Brief accent flash, then settle |

**Two states, one element (CLOUD-05):**
1. **Full-screen** — at boot and whenever KERNEL is speaking/listening. The cloud owns the whole canvas; widgets bloom from it.
2. **Corner pill** — during a Claude Code session, the cloud shrinks and migrates to the **top-left** corner as a compact pill carrying the live, scrollable Kernel↔Claude transcript. The transition between full-screen and pill is a spring (nothing snaps — see Motion Law); the pill retains a miniature live cloud + an accent live-pulse dot while streaming.

## Motion Law (domain extension — load-bearing)

**The one rule: nothing ever snaps. Everything eases, drifts, settles. Numbers count up.** CLOUD-06, VOICE-04.

| Motion | Spring feel (response / damping) | Timing | Notes |
|--------|----------------------------------|--------|-------|
| Widget bloom (present) | response ≈ **0.5s**, damping ≈ **0.8** | ~450–600ms perceived | scale 0.96→1.0, opacity 0→1, forward-blur clears to 0 |
| Widget dissolve (dismiss) | response ≈ **0.45s**, damping ≈ **0.85** | ~400–550ms | scale 1.0→0.96, opacity 1→0, blur returns as it recedes into the cloud |
| Cloud full-screen ↔ corner pill | response ≈ **0.6s**, damping ≈ **0.8** | ~600ms | spring migration + scale-down; no snap, no hard cut |
| Mic-amplitude particle push | continuous, frame-rate driven (not a spring) | per-frame | RMS-mapped outward force; smoothed so it breathes, not jitters |
| TTS boundary burst | response ≈ **0.3s**, damping ≈ **0.7** | ~250–350ms | localized burst + brighten flash, quick settle |
| Number count-up | ease-out, **0.6–0.9s** to final value | — | balances, counts, chart totals count up from 0 (or from prior value); tabular numerals keep width stable |
| Focus/active ring | response ≈ **0.35s**, damping ≈ **0.9** | ~300ms | gentle fade-in of the accent hairline/glow |

**Hard prohibitions:** no instantaneous appearance/disappearance of any widget; no linear cuts between cloud states; no opacity/position step changes. If a callback is missed, the fallback path (below) must still ease, never snap into place.

## Stage Choreography (domain extension — the differentiator)

A `Stage` controller, driven by the routine engine and TTS boundaries, makes widgets bloom forward when their topic is narrated, hold while spoken about, then disperse back into the cloud. CLOUD-04, VOICE-03, VOICE-04.

**Choreography contract:**
- **One or two widgets in focus at a time.** Content follows attention. Never a wall of cards (the explicitly-discarded anti-feature). When a third topic arrives, an existing widget must dissolve before/as the new one blooms.
- **Bloom = forward from the cloud:** particles coalesce → glass widget materializes at scale 0.96→1.0, opacity in, forward-blur clearing (per Motion Law bloom spring).
- **Hold:** widget stays in sharp focus (blur 0, full opacity) for the duration the topic is being spoken about.
- **Disperse:** on topic end, the widget recedes (scale→0.96, opacity out, blur returns) and visually dissolves back into the cloud — particles reabsorb.

**Sync mechanism (the contract the daemon + Face must honor):**
- The daemon sends `speak` frames carrying **character-keyed `cues[]`** (character offsets into the utterance + which widget to present/dismiss). **The daemon NEVER sends timing-estimate messages** — it sends character offsets; the Face owns the clock.
- The Face's AVSpeechSynthesizer `willSpeakRangeOfSpeechString` delegate emits word/segment boundaries; on each **boundary crossing past a cue's character offset**, the Face fires `Stage.present(widget)` / `Stage.dismiss()` and the matching particle burst.
- **Dual-pacing resilience (VOICE-04):** the Stage controller MUST support both:
  1. **Word-level (callback-driven)** — fires on real boundary callbacks (preferred, tightest sync).
  2. **Sentence-level (time-based fallback)** — if boundary callbacks are flaky/missing on the target macOS+voice, the Stage falls back to sentence-duration pacing so choreography degrades gracefully but **still eases/blooms — never snaps and never stalls a widget mid-screen.**
- Guard against AVSpeechSynthesizer's known out-of-bounds `NSRange` quirk: an invalid range must not crash or freeze the Stage — it falls through to the time-based path.

**Phase 3 delivery bar:** at minimum, **ONE widget choreographed end-to-end** (bloom on narration → hold → dissolve), proving the talk→reason→speak→choreograph loop. The visual contract for all five widgets is specified below so Phase 4 renders the rest against a fixed contract.

## Per-Widget Visual Specs (domain extension)

All five widgets are **frosted glass, bloom-and-dissolve, 16–20px corner radius, 24px (lg) interior padding, 1px white-7% hairline border**. Headings use Heading (20/600); body uses Body (16/400); muted metadata uses Label (14/400, zinc-400). Money/counts always tabular. Accent appears only per the reserved list. Each widget specifies its empty/error states.

### 1. events
- **Content:** count headline ("3 events" — Display 28/600, count counts up), then a vertical list of up to ~3 events: time (tabular, Label) + title (Body) + optional location (Label muted).
- **Layout:** count headline top; list below at 8px (sm) row gap; time column left-aligned with tabular numerals so times align.
- **Empty:** heading "No events today" / body "Your calendar's clear." (no accent, no CTA in P3).
- **Error:** "Couldn't reach the calendar." / "KERNEL will retry on the next brief."

### 2. mail (with suggested-action chips)
- **Content:** unread count headline (tabular, counts up), then up to ~3 message rows: sender (Body semibold) + subject snippet (Label muted, single line truncated). Below each (or the focused) message: **suggested-action chips** — `Reply` · `Open` · `Archive` · `Log` (28px min height, 12px horizontal padding, neutral glass fill, hairline border; the currently-selected/active chip carries the accent focus ring — accent reserved use #2).
- **Layout:** count top; rows at sm gap; chips in a horizontal wrap at xs gap beneath the focused row.
- **Empty:** "Inbox clear." / "Nothing needs you right now."
- **Error:** "Couldn't read Mail." / "Check Mail is open; KERNEL will retry."

### 3. accounts (balances)
- **Content:** one row per account: account name (Body) + balance (Display 28/600 **tabular**, counts up). Multiple accounts stack with a total at bottom (Heading + tabular).
- **Layout:** name left, balance right-aligned with tabular numerals so decimal points align across rows; lg interior padding; sm row gap.
- **Empty:** "No accounts linked." / "Link an account to see balances." (link flow is Phase 4 — no live CTA in P3).
- **Error:** "Balances unavailable." / "KERNEL couldn't reach the aggregation service."

### 4. spending (W/M/Y chart)
- **Content:** total spend headline (Display, tabular, counts up) + a bar/line chart + a **W / M / Y timeframe switcher**. Active timeframe segment shown in accent text/underline (accent reserved use #4) — the other two neutral. Axis ticks tabular.
- **Layout:** headline top-left, switcher top-right (segmented, neutral with one accent-marked active label); chart fills the body with lg padding; chart redraws with an ease (not a snap) when timeframe changes; bars/values count up on first bloom.
- **Empty:** "No spending yet." / "Transactions will appear once an account syncs."
- **Error:** "Spending data unavailable." / "KERNEL couldn't aggregate transactions."

### 5. email-preview ("Send it?" card)
- **Content:** the highest-stakes widget. Labeled fields stacked: **To** / **Subject** / **body** / **signature** — each label in Label (14/400 muted), each value in Body (16/400). The body field is the tallest. Footer holds the gate action: **"Send it?"** with **Send** (primary) and **Edit** (secondary). In Phase 3 this widget's visual contract is specified but the live send/gate flow ships in Phase 4 — render read-only/preview here.
- **Layout:** `.regularMaterial` (denser, for text legibility); field labels left-aligned, values beneath at xs gap; fields separated by sm gap; footer actions right-aligned at md gap above.
- **Accent:** Send button carries the accent focus ring when focused (reserved use #2); it is NOT accent-filled at rest.
- **Destructive note:** sending is a Yellow-tier action in Phase 4 — the **explicit "Send it?" + Send tap IS the gate**; never auto-send; the To field, if it came from external content, must be visibly shown before send (Phase 4 enforcement, but the design surfaces To prominently so this is honored).
- **Empty:** n/a (only appears when a reply is composed).
- **Error:** "Couldn't draft this reply." / "Tell KERNEL the intent again."

## Copywriting Contract

KERNEL's owner-facing voice is **direct, terse, reporting-style — no filler, no chatty preamble** (PERS-01). Copy below follows that register.

| Element | Copy |
|---------|------|
| Primary CTA | **Send it?** (email-preview widget — the only explicit action surface in this phase; Phase 4 wires the gate) |
| Empty state heading | Per-widget (see specs): "No events today" / "Inbox clear." / "No accounts linked." / "No spending yet." |
| Empty state body | Per-widget next-step line (see specs), e.g. "Your calendar's clear." |
| Error state | "Couldn't reach the calendar." → "KERNEL will retry on the next brief." (pattern: terse problem + what KERNEL does next) |
| Destructive confirmation | Send (Yellow-tier, Phase 4): "Send it?" — explicit Send tap is the confirmation; Edit returns to preview. No destructive/irreversible action ships in Phase 3. |

**Voice rules for all UI copy:** no exclamation marks beyond the rare warranted one; no "Oops!"/"Uh-oh"; no apologetic padding; state the fact and the next step. Counts and money read as plain figures ("3 events", "$1,240.00"), never spelled out.

## 6-Pillar Quality Gate (acceptance per dimension)

Final visual fidelity on a living, GPU-rendered, motion-choreographed interface **cannot be fully verified statically**. The gate therefore splits into **structural conformance (verifiable)** and **visual fidelity (MANUAL owner check — the app must be run and watched)**.

| # | Dimension | Acceptance (structural — verifiable) | Acceptance (fidelity — MANUAL owner check) |
|---|-----------|--------------------------------------|--------------------------------------------|
| 1 | Visual hierarchy | One or two widgets in focus max; out-of-focus surfaces depth-blurred + dimmed; count/headline uses Display, metadata uses Label | Owner runs app: attention lands on the spoken topic; never a wall of cards |
| 2 | Consistency | All widgets use the same radius (16–20), padding (24/lg), hairline border (white 7%), material, type roles | Owner watches a brief: widgets feel like one family blooming from one cloud |
| 3 | Spacing / alignment | 4-pt grid honored; tabular numerals align money/times in columns; declared exceptions only | Owner check: no ragged columns, no off-grid drift while widgets bloom/settle |
| 4 | Typography | Exactly 4 sizes, 2 weights present; SF Pro Display/Text split correct; tabular numerals on all money/counts | Owner check: numbers don't reflow width while counting up |
| 5 | Color / contrast | One accent only, used only per the reserved list; near-black base + zinc neutrals; hairline borders at 6–8% | Owner check: AA legibility of body/muted text over live glass on the moving cloud (the dynamic background is why this is manual) |
| 6 | Motion | Spring constants present per Motion Law table; no instantaneous show/hide in code; dual-pacing fallback wired | **Primary manual check:** owner runs and watches — nothing snaps; cloud breathes with voice; widgets bloom/dissolve in sync with words; count-ups ease |

**Structural conformance is checkable by:** tokens/constants present, the five widget views exist, the two cloud states are wired, the Stage controller exposes present/dismiss + both pacing modes, accent usage greppable to the reserved list.

**Explicit call-out:** dimensions tied to motion, the live cloud's amplitude reaction, boundary-synced choreography, and contrast-over-moving-background are **NOT signed off by static inspection** — the owner MUST run the Face, speak to KERNEL, and watch the cloud react and a widget choreograph before this contract is marked visually satisfied. The checker verifies structure; the owner verifies fidelity.
