<objective>
Build the email reply flow: on "reply", KERNEL takes a one-line intent, ALWAYS injects the ~200-token voice profile (`kernel-memory/knowledge/voice-profile.md`), pulls 2-3 few-shot past emails most similar to the recipient via the shipped `retrieveAndRerank`, routes stakes (casual → 7B helper, high-stakes → cloud `ClaudeBrain`), and produces a preview-card payload. NOTHING sends without an explicit `ui.intent{intent:'send-email'}` from the Face Send control — the preview + explicit Send IS the Yellow gate (04-RESEARCH Pattern 4). On send, the `mail` tool dispatches through `registry.dispatch → gate.authorize` (Yellow), drives Peekaboo Mail, marks the source read, and logs. KERNEL never auto-sends and never sends to an externally-sourced To address without showing it.

The EmailPreviewWidget file was created in 04-01 as the visual contract; this plan wires its Send path to the daemon send flow (the widget still emits only the intent — the daemon executes the gated send). Default send path is Peekaboo Mail.app (locked decision); the tool is structured so a Gmail path could be added later.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Inbound email / few-shot corpus → daemon | Email bodies + recipient text are `source:external` untrusted content |
| Reply To address (external-sourced) → send | An externally-sourced recipient must be surfaced before any send |
| Face Send intent → daemon dispatch | The only path to a send crosses here, and only after explicit user action |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-12 | Tampering | auto-send / send-to-external abuse | mitigate | explicit Send ui.intent is the only send path; never auto-send; external-sourced To shown before send (MAIL-04/05) |
| T-04-13 | Elevation of Privilege | indirect prompt injection via email body / few-shot | mitigate | profile + few-shot injected as DATA/examples, never instructions; the To, if external, is surfaced; no external content drives a Red action |
| T-04-14 | Elevation of Privilege | memory poisoning via auto-promoted email "fact" | mitigate | voice-profile is a human-reviewed durable promotion; no email body auto-writes to knowledge/IDENTITY (Pitfall 4) |
| T-04-15 | Tampering | send bypassing the gate | mitigate | the mail send op reaches registry.dispatch → gate.authorize (Yellow); compose never sends; anti-bypass contract honored |
| T-04-16 | Information Disclosure | logging full email content | mitigate | pino logs the send event metadata, not full body content (ASVS V7) |
</threat_model>
