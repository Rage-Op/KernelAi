<objective>
Build the Phase-2 foundation: the tool router and the single `gate.authorize` chokepoint that every tool dispatch passes through. This plan fills two RESERVED Phase-1 seams without rewriting them â€” the `loop.ts` act no-op (`if (decision.action) { /* P2+: router.dispatch */ }`) and the reserved `safety/` gate (currently a README contract). It defines the contracts the two real tool adapters (Plans 02-02 Peekaboo, 02-03 browser) register against.

The gate is CLASSIFY-ONLY here: it labels a call Green/Yellow/Red and enforces the one hard rule whose physical capability lands this phase â€” the credential fence â€” and records (logs) the tier. It does NOT run a circuit breaker, `/override`, dry-run, or spend ceiling (all gated to Phase 5). The `Verdict` shape is designed so Phase 5 enables `gated` behavior INSIDE `authorize` WITHOUT touching the router, the tools, or the loop.

LOCKED DECISION (research Open Question 1): a Red-tier classification in Phase 2 = **deny + escalate** (no Red autonomy before Phase 5). The credential fence is ALWAYS a hard deny regardless of tier.

Purpose: Establish the access-control boundary (ASVS V4: default-deny) and the input-validation boundary (ASVS V5: zod-validate every ToolCall.args) before any tool can act. No tool self-classifies; no path bypasses the chokepoint.
